Last year, I tried to reduce API latency and reached for Redis and EC2 auto scaling too early.

It felt responsible. I had seen the pattern everywhere: cache slow work, add compute when traffic rises, and call the system scalable.

The tools were not the problem. The mistake was reaching for them before the bottleneck justified their cost.

<!--truncate-->

The real bottleneck was smaller: the service repeated the same expensive calculation inside one process. I did not need shared infrastructure yet. I needed to stop doing redundant work.

In Python, that can be as small as this:

```python
from functools import lru_cache

@lru_cache(maxsize=128)
def compute_price_rules(account_id: str) -> dict:
    return load_and_parse_rules(account_id)
```

That local cache would have solved most of the latency problem without a new server, credential, network hop, deployment path, or on-call concern.

The lesson is not that `@lru_cache` is better than Redis. The lesson is that every scaling tool buys something specific and charges somewhere else.

Local caching is cheap and fast, but it is process-local. It does not share state across machines, and it disappears on restart.

Redis gives you shared state, explicit TTLs, and cross-process coordination. It also adds a dependency, network latency, access control, backup questions, memory pressure, and cache invalidation risk.

Auto scaling gives you more capacity. It also makes deploys, cold starts, state placement, observability, and cost control harder to ignore.

Those costs are not reasons to avoid mature infrastructure. They are reasons to be precise about the problem.

Before adding infrastructure, I ask:

- Is the slow work repeated inside one process?
- Is the data safe to cache?
- How stale can the answer be?
- What invalidates it?
- Does the cache need to be shared across instances?
- What happens when the cache is empty, wrong, full, or unavailable?
- Will this change make the system easier to operate, or only more impressive to describe?

These questions matter in startups because time is scarce. They matter in enterprise systems because every component becomes part of the control plane, the security surface, and the incident story.

Premature scaling is dangerous because it can look like good engineering. You are using serious tools, thinking about growth, and doing work that sounds production-minded.

But production engineering is not adding production-shaped components. It is choosing the smallest mechanism that preserves the right invariant under the expected pressure.

In this case, the invariant was simple:

> Repeated requests should not recompute stable data unnecessarily.

The smallest useful mechanism was a bounded in-process cache.

Redis would have been appropriate later if I needed shared cache state, coordinated invalidation, or consistent behavior across workers. Auto scaling would have been appropriate later if one process could no longer handle measured load.

So the rule I use now is simple:

Start by removing waste. Add infrastructure when the remaining problem genuinely needs infrastructure.

That is usually faster, safer, and easier for future operators to debug at 3 a.m.

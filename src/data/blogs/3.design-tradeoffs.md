Good interface design is not about always choosing the cleanest abstraction.

It is about knowing which constraint matters most.

In many application APIs, flexibility is the right default. A single function that accepts a list, an options object, or a generic configuration can make a system easier to use and easier to change.

But some systems have different priorities. When execution cost, deterministic behavior, auditability, and security boundaries matter more than developer convenience, a flexible interface can become the expensive choice.

The Ethereum Virtual Machine gives a small but useful example: `LOG0` to `LOG4`.

<!--truncate-->

## The EVM Example

The EVM uses `LOG` opcodes to emit events. These events are not stored as contract state, but off-chain clients can read and filter them through transaction logs.

Each log has:

- a payload, read from memory
- zero to four indexed topics, used for filtering

Instead of one dynamic opcode like `LOG(topics[])`, the EVM defines five separate opcodes:

- `LOG0`: emits a log with 0 indexed topics
- `LOG1`: emits a log with 1 indexed topic
- `LOG2`: emits a log with 2 indexed topics
- `LOG3`: emits a log with 3 indexed topics
- `LOG4`: emits a log with 4 indexed topics

At first glance, this can look less elegant than a single generic instruction.

That reaction is reasonable. In a higher-level language, many of us would write something like:

```ts
log(topics: Topic[], data: Bytes)
```

For an application API, that may be the better design.

For a consensus VM, it creates different problems.

## What the Fixed Opcodes Buy

### 1. A Stable Stack Shape

The EVM is stack based. Each opcode consumes a specific number of stack items.

With `LOG0` to `LOG4`, the VM knows the stack shape from the opcode itself. `LOG0` consumes memory position and size. `LOG4` consumes those plus four topics.

That makes execution easier to implement, inspect, and reason about. There is no extra topic count to decode, no dynamic topic array to parse, and fewer invalid shapes to reject at runtime.

The gas calculation still depends on payload size, memory expansion, and topic count. The important point is not that every log has one flat cost. The point is that the operation shape is explicit.

In a gas-metered system, explicit shape matters.

### 2. Less Runtime Ambiguity

A dynamic design would need more rules:

- where the topic array lives
- how its length is encoded
- what happens when the length is too large
- how bounds are checked
- how memory expansion is charged
- how malformed inputs fail

Each rule may be manageable by itself. Together, they add surface area.

In ordinary product code, this may be acceptable. In a blockchain VM, every extra rule becomes something all clients must implement exactly the same way forever.

That is a high bar.

### 3. Better Consensus Safety

Ethereum nodes independently execute the same transactions and must agree on the same result.

That makes determinism a security property, not just an implementation detail. If two clients interpret an operation differently, the risk is not a local bug. It can become a consensus failure.

Fixed opcodes reduce the number of interpretation paths. They make the VM less flexible, but also easier to specify, test, and reproduce across implementations.

That is the tradeoff.

## The Tradeoff

The EVM chose predictability over convenience.

It gave up a more general interface so the VM could keep a smaller, clearer execution model. That is not because generic APIs are bad. It is because this layer has to optimize for different things:

- deterministic execution
- simple gas accounting
- small implementation surface
- clear failure behavior
- long-term compatibility across many clients

A higher-level system might choose differently.

For example, a web application SDK probably should not expose five separate methods if one typed method is safer and easier for developers to use. In that context, a small amount of internal complexity can be worth it if it prevents misuse.

The mistake is treating one style as universally mature.

Sometimes abstraction reduces risk. Sometimes it hides risk.

## What API Designers Can Take From This

When designing an interface, the useful question is not:

> What is the cleanest abstraction?

The better question is:

> What must remain predictable when this interface is used under pressure?

If the answer involves money, consensus, authorization, audit logs, real-time latency, hardware, or irreversible side effects, explicit interfaces may be safer than flexible ones.

That can mean:

- choosing a few narrow operations instead of one general operation
- making limits visible in the API shape
- avoiding runtime interpretation when compile-time shape is enough
- preferring boring failure modes over clever behavior
- documenting what was optimized and what was deliberately not optimized

This is the same reason many low-level systems prefer fixed schemas, fixed syscall shapes, explicit wire formats, and narrow protocol messages. The goal is not to make the interface feel clever. The goal is to make behavior inspectable and hard to misunderstand.

## The Practical Lesson

In startups, flexible APIs can help teams move quickly while the product is still changing.

In enterprise and security-sensitive systems, the same flexibility can become a maintenance and incident cost if it hides important behavior.

Neither instinct is wrong. They are answers to different operating conditions.

The senior engineering move is to name the constraint before choosing the design.

If change is the main risk, design for flexibility.

If ambiguity is the main risk, design for explicitness.

If failure must be reproduced exactly across machines and years, keep the interface small, bounded, and boring.

That is what `LOG0` to `LOG4` demonstrates. It is not just an odd EVM detail. It is a reminder that good design is not the absence of tradeoffs.

Good design is knowing which tradeoff you are making, and why that tradeoff is safe for the layer you are working in.

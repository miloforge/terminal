# Cutting AI Tokens Without Moving Risk

In startups, optimization is not always polish. Sometimes it is how the system stays affordable enough to keep running.

I replaced a legacy decision point with a Gen AI task. The result was good enough to use, but the cost profile was wrong for the volume. At peak, the workflow processed millions of messages, and this model call sat in a hot path.

The first version used about 89 tokens per item:

```txt
Input tokens: 78
Output tokens: 11
Total tokens: 89
```

The model was `gemini-2.5-flash-lite` at `$0.05 / 1M` tokens. That sounds small until it runs across millions of items every day.

<!--truncate-->

## The tempting fix

The obvious move was batching. It offered a 50% discount, but it came with a practical problem: results could take up to 24 hours.

The workflow already had a daily delivery promise:

```txt
Input data -> preliminary processing -> hourly aggregation -> decision -> daily post-processing -> delivery
```

Moving the decision into a next-day batch would save money, but it would also put the delivery window at risk. That is not a clean optimization. It moves cost out of the invoice and into operational uncertainty.

## The smaller change

Instead of batching the whole day, I batched by hour and compressed the prompt.

That kept the decision close to the existing workflow while reducing repeated prompt overhead. For a batch of 50 items, the compressed path used:

```txt
Input tokens: 975
Output tokens: 399
Total tokens: 1374
```

The comparison was straightforward:

```txt
Original path: 89 * 50 = 4450 tokens
Hourly batch + compression: 1374 tokens
```

That is about 70% fewer tokens in the hot path.

## What could go wrong

Prompt compression is not just a cost change. It changes the shape of the input, so it also changes the privacy boundary, the decision boundary, and the blast radius of a bad response.

The invariant I cared about was simple:

> Compression may remove tokens, but it must not remove the facts needed to make the same business decision safely.

That creates real failure modes:

- A larger batch can expose more user or business data in one request.
- One item can influence another if item boundaries are weak.
- Compression can drop details that look redundant but change the decision.
- The cheaper path can drift silently while still returning valid-looking output.
- A malformed item or provider failure can block many decisions at once.
- If only the compressed prompt is retained, later audit becomes weak.

Those are not theoretical concerns. In enterprise systems, they are the difference between "the model was cheaper" and "we can still explain, contain, and recover from the model's behavior."

## The controls

I would not roll this out just because the token count improved. The compressed path needs guardrails:

- redact or minimize sensitive fields before compression
- require one output for each stable item ID
- keep strict item separators so records cannot merge
- compare a deterministic sample against the uncompressed path
- track disagreement between original and compressed decisions
- alert on missing IDs, duplicate IDs, output shifts, and abnormal rejection rates
- cap batch size so one failure does not affect the whole day
- retry failed items individually instead of replaying the whole batch
- keep enough audit data to explain the decision without storing raw sensitive input forever

The acceptance test is not "the prompt is shorter."

It is:

> The compressed path must match the original decision often enough for the business risk, and every disagreement must be measurable, attributable, and recoverable.

That means shadow traffic first: run a controlled sample through both paths, inspect disagreements, set thresholds, and only then move traffic gradually.

## Why it mattered

The win was not only lower cost. The win was keeping the same decision inside the same delivery promise.

The result was a better operating shape:

- about 70% fewer tokens
- no next-day dependency
- no downstream rewrite
- explicit privacy and correctness risks
- a rollout path that could catch drift before users felt it

Good optimization does not hide risk. It makes the system cheaper while keeping the important boundaries visible.
